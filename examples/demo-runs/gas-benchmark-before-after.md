# Gas benchmark ŌĆö before / after (reproducible)

**Real, measured gas numbers** comparing an intentionally unoptimized baseline vault against
the optimized `TokenVault` that Mantle Forge scaffolds and deploys. Every number below comes
from transaction receipts on the Hardhat EVM ŌĆö nothing here is estimated or hand-written.

> This is the evidence behind the `mantle-gas-analysis` skill's "before/after" claim: the skill
> doesn't just *describe* optimization, this artifact **demonstrates** it with numbers anyone can
> reproduce.

## How to reproduce (Ōēł3 s)

```bash
cd templates/token-vault
npm install
npx hardhat test --grep "gas benchmark"
```

The test ([`test/gas-benchmark.ts`](../../templates/token-vault/test/gas-benchmark.ts)) deploys both
contracts, runs `deposit()` and `withdraw()` on each, reads `gasUsed` from the receipts, prints the
table below, and **asserts** the optimized version is cheaper on every operation (the test fails if
it ever regresses).

## Results

| Operation | Before (baseline) | After (optimized) | Gas saved | Savings |
|-----------|------------------:|------------------:|----------:|--------:|
| Deployment | 677,107 | 624,044 | 53,063 | **7.83%** |
| `deposit()` | 81,129 | 78,743 | 2,386 | **2.94%** |
| `withdraw()` | 67,992 | 65,444 | 2,548 | **3.74%** |

Environment: Solidity `0.8.24`, optimizer enabled (200 runs), Hardhat in-process EVM (Paris).
Identical happy-path behaviour in both contracts ŌĆö only the implementation patterns differ.

## What changed, and why each line saves gas

Compared contracts:
[`TokenVaultBaseline.sol`](../../templates/token-vault/contracts/TokenVaultBaseline.sol) (before) vs
[`TokenVault.sol`](../../templates/token-vault/contracts/TokenVault.sol) (after).

| Optimization | Anti-pattern (before) | Optimized (after) | Effect |
|--------------|-----------------------|-------------------|--------|
| `immutable` state | `address owner; IERC20 asset;` in storage | `immutable owner` / `immutable asset` | Removes an `SLOAD` (~2,100 gas cold) on each access and shrinks deploy bytecode |
| Custom errors | `require(cond, "TokenVault: ŌĆ”")` | `revert ZeroAmount()` etc. | Removes revert strings from bytecode ŌåÆ smaller deploy + cheaper revert paths |
| `unchecked` math | checked `balance - amount` after a `>=` guard | `unchecked { balance - amount }` | Skips a redundant overflow check the guard already proved safe |
| Cached reads | `_balances[msg.sender]` read 2├Ś from storage | cached in a memory local | Removes a redundant warm `SLOAD` in `withdraw()` |

## Honest reading of the numbers

Runtime savings (`deposit`/`withdraw`) are intentionally **modest (2ŌĆō4%)** because both versions
share the dominant costs ŌĆö the `SafeERC20` token transfers, balance lookups, and `ReentrancyGuard`
SSTOREs ŌĆö which are required for correctness and are *not* optimized away. The larger win is at
**deployment (7.83%)**, driven by `immutable` state and custom errors cutting bytecode size. Mantle
Forge prefers **safe, readable** optimizations over risky micro-opts, so it does not chase the
shared-cost lines at the expense of security.
